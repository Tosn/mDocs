# mDocs 开发规则（Rules）

> 项目：桌面端文档查看 / 搜索 / AI 问答工具。
> 配套：`feature/spec.md`（需求）、`feature/plan.md`（方案）、`feature/tasks.md`（原子任务）。
> 本文件是开发约定的唯一来源；`CLAUDE.md` 只做指向。

## A. 工作流程

1. **RIPER 协议（全局强制）**：每次回复开头声明 `[模式：名称]`；仅当收到独立成行的 `RESEARCH/INNOVATE/PLAN/EXECUTE/REVIEW` 才切换模式。
   - EXECUTE 只实现**已批准计划**中明确列出的内容；发现需偏离，立即停下退回 PLAN 说明原因。
   - 极小改动（拼写、格式）可在征得同意后简化流程。
2. **以 spec 为准绳**：每项实现都对应 `spec.md` 的验收标准；提交前对照核对。
3. **改动需求/方案先改文档**：任何范围变化先更新 `feature/*`，再动代码，保持三份文档一致。

## B. 编码纪律

4. **测试先行（TDD）**：按 `tasks.md` 顺序推进，**奇数任务写测试、偶数任务写实现**，先红后绿。
4.1 **严格 TDD 红绿顺序（强制）**：先写测试，并**实际运行确认它失败（红）**——若测试一上来就通过，说明测试无效或实现已存在，须停下排查，不得跳过；再写实现，**只为让该测试通过（绿）**，不超出测试覆盖范围做额外改动。每对任务必须经历「红 → 绿」两步且各自运行验证。
5. **单文件原则**：每个任务**只改一个文件**；测试文件与实现文件分属相邻两个任务，实现任务不得改测试。
6. **任务状态维护**：完成任务后把 `tasks.md` 对应行行首 `⬜` 改为 `✅`，并更新文末「进度概览」计数。
7. **测试框架**：Vitest；React 组件用 `@testing-library/react`；主进程用内存 SQLite，对 safeStorage / 网络 / 嵌入 API / LLM 一律打桩。
8. **命令前缀**：所有 shell 命令用 `rtk` 前缀（见全局 RTK 约定），命令链中每段都加。
8.1 **前端样式（强制）**：所有涉及前端样式的工作（新增/修改 UI、组件外观、CSS/Tailwind、布局、配色、排版、交互态、动效等）**必须先调用 `/ui-ux-pro-max:ui-ux-pro-max` 技能**，依其设计指引产出样式，不得凭空手写。

## C. 架构边界

9. **主/渲染隔离**：渲染进程（`src/`）只通过 `window.api` 通信，**不直接访问文件系统与密钥**；特权操作放主进程（`electron/`）。
10. **统一返回**：所有 IPC 返回 `Result<T> = { ok:true; data } | { ok:false; error:{code,message} }`；通道名集中在 `shared/channels.ts`。
11. **不引入 spec「非目标」**：多人协作、内容版本回溯、本地大模型问答、md/txt/pdf/网页 以外格式 —— 一律不做。

## D. 必须守住的产品不变量（违反即视为缺陷）

- **离线降级**：无网 / AI 不可用时，查看、目录文件增删改、关键词搜索照常；仅问答不可用并明确提示原因。
- **回答可信**：AI 回答须标来源且可点击跳转（`MessageSource`）；文档无答案时如实回「未在文档中找到相关内容」，不臆造；空库提问提示「请先添加文档」。
- **回收站**：删除须二次确认 → 入回收站保留 **7 天**可恢复；过期项在**下次启动**时由 `startup/trash-gc` 彻底删除；恢复时原父级缺失则落默认位置并告知。
- **隐私与密钥**：API Key 存 `safeStorage`，**不入明文 DB**；首次使用外部模型须隐私告知（内容会发送给厂商）。
- **模型记忆**：默认沿用上次所用模型；手动切换时弹 Key 输入框，已存过则**回填掩码**。
- **网页爬取**：抽正文转 md 并**保留图片等富媒体**、留存原链接；失败给明确错误码、**不落空文档**。
- **上传入口**：统一支持文件或文件夹（文件夹连同内容按层级**递归上传**、跳过不支持格式并汇总）。
